04 The decision rule

Pick by the question you’re answering.

PostHog

“We want a live test this week.”

Why

One afternoon to a working test; editors run everything from the Payload admin tab.

Key risk

Plugin repo went private; 2-arm tests only.

Mitigation

Vendor the plugin source from npm; fork if it stalls.

Statsig

“We’re building an experimentation program.”

Why

Strongest stats engine (CUPED, sequential, holdouts) for hundreds of concurrent tests.

Key risk

Most build effort; event data lives in Statsig’s cloud.

Mitigation

Glue mirrors Statsig’s documented Contentful blueprint.

GrowthBook

“The lakehouse stays the source of truth.”

Why

Warehouse-native results, open source, n-way tests, zero network on the page view.

Key risk

Scorecards need a warehouse connection and defined metrics.

Mitigation

Start on GrowthBook Cloud; self-host later, nothing changes.

05 Decide, then do this

The recommendation.

Talk to all three.

Book demos with PostHog, Statsig and GrowthBook. Walk each vendor through the Payload setup, the Phase One CTA, and the 170-ZIP rollout — then pick by the condition that matters most.

If the priority is a Payload-native integration → PostHog. The only one with an existing payload-ab plugin; editors run the test from the admin tab.

If the priority is experimentation rigor → Statsig. Strongest stats engine (CUPED, sequential, holdouts) for a real program.

If the priority is lakehouse-native results → GrowthBook. Open source, queries Fabric directly, zero network on the page view.
